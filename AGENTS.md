# 專案確認機制

## Master Rule: Confirmation Protocol

在此專案中，不論使用者提出什麼要求，尤其是涉及程式碼修改時，必須嚴格遵守以下流程：

1. 摘要理解

   先用 3 到 5 個重點摘要對需求的理解，內容必須包含：

   - 這次修改的目標是什麼。
   - 哪些功能要保留，例如手動計算邏輯。
   - 哪些限制要遵守，例如禁止將數據搬回前端。

2. 執行計畫

   簡述打算如何修改，例如會修改哪個 function、component、API route、CSS 區塊或設定檔。

3. 暫停並詢問

   在輸出任何程式碼或進行程式碼修改之前，必須先詢問使用者：

   「以上理解與計畫是否正確？若正確請告知，我將開始執行。」

4. 程式碼輸出規則

   - 如果需要修改的 code 小於 4 處，輸出修改位置與內容，讓使用者手動修改。
   - 如果需要修改的 code 超過 4 處，必須先和使用者確認是否要輸出大段 code。
   - 除非使用者明確回覆「正確」或「開始」，否則禁止直接輸出大段程式碼。
